# G-Designer / ClaimFlow 小白代码阅读指南

## 建议阅读顺序

1. 先看数据如何变形：`datasets/gsm8k_dataset.py`、`datasets/mmlu_dataset.py`。
2. 再看实验如何取题：`experiments/run_gsm8k.py`、`run_humaneval.py`、`run_mmlu.py`。
3. 看 `GDesigner/graph/graph.py`，理解如何生成节点、计算边概率和安排执行顺序。
4. 看 `GDesigner/graph/node.py`，理解一个 Agent 从前驱收到了什么数据。
5. 看 `GDesigner/agents/`，理解题目和其他 Agent 输出如何拼成 Prompt。
6. 看 `GDesigner/llm/gpt_chat.py`，这里的 `achat` 才是真正发 HTTP 请求调用大模型。
7. 看 `agents/final_decision.py`，理解原方法如何把所有 Agent 输出汇总成最终答案。
8. 最后看 `GDesigner/claimflow/runtime.py`，对比声明级路由如何替代整段答案传播。

## 最常见的数据形态

| 阶段 | Python 形态 | 含义 |
|---|---|---|
| 数据集记录 | `dict` / `pandas.Series` | 原题、选项、代码测试和金标 |
| Graph 输入 | `{"task": "..."}` | Agent 唯一必须看到的原题 |
| Agent Prompt | `[system消息, user消息]` | 角色约束、原题和允许接收的通信内容 |
| Agent 输出 | `str` | 自然语言推理、选项或 Python 代码块 |
| Node 输出 | `list[str]` | Node 为兼容多结果统一使用列表保存 |
| 原 Graph 边概率 | `Tensor[N*N]` | 每条 source->target 候选边一个 logit/概率 |
| ClaimFlow 输出 | `ClaimBundle` | 候选答案、假设、原子声明和原始文本 |
| ClaimFlow 状态 | `ClaimStore` | 声明、分支、Agent-Claim 边和 Claim-Claim 边 |
| 最终日志 | JSON / JSONL | 最终答案、正确性、成本和完整中间轨迹 |

## 图概率采样只记住这五步

```text
问题/角色文字
  -> 语义向量
  -> GCN + MLP 得到每个 Agent 的向量
  -> 两两点积得到 edge_logit
  -> p = sigmoid(edge_logit / temperature)
  -> 随机数 u < p 时建立边
```

建立边记录 `log(p)`，没建立记录 `log(1-p)`。训练时把这些值相加为 `log_prob`，
再与答案奖励相乘。`temperature` 越大，概率越靠近 0.5，采样越随机。

ClaimFlow 的策略概率不同：它依次对“声明、操作、目标 Agent”建立三个分类分布，
三个 log probability 相加就是一个完整动作的 log probability。

## 原方法与 ClaimFlow 的核心区别

原方法的通信单位是整个 `str` 回答，最终节点通常还会调用一次 LLM 重新汇总。
ClaimFlow 的通信单位是 `ClaimNode`，控制器一次只处理一条声明，并允许验证、质疑、
分叉、隔离和证据合并。已验证候选直接从 `BranchState.exact_output` 返回，不再重写。
